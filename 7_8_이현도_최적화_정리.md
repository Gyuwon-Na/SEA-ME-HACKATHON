## 1. 카메라 노드 CPU 최적화 (MJPG passthrough)

### 문제
`onboard.launch` 실행 중 htop 확인 결과 `camera_node`가 **코어 하나를 통째로(~105%)** 잡아먹음.
단순히 30fps 압축 이미지를 퍼블리시하는 노드가 이럴 이유가 없어서 원인 분석.

### 원인 — 디코딩 → 재인코딩 → 디코딩 왕복
USB캠(640×480, MJPG)이라 프레임 하나가 아래 경로를 탐:

1. USB캠 → **MJPG 전송**
2. GStreamer `jpegdec` → MJPG **디코딩** (CPU)
3. `videoconvert`/`videoscale` → BGR 변환 (CPU)
4. 콜백에서 `cv2.imencode('.jpg', quality=90)` → **다시 JPEG 인코딩** (CPU)
5. `bisa_autonomous_node`가 받아서 → **또 JPEG 디코딩** (CPU)

즉 같은 칩에서 **디코딩→인코딩→디코딩**을 다 함.
`camera_node`는 그 프레임을 쓰지도 않고 그냥 재인코딩만 하는데도 코어 하나를 소모.

### 이 구조가 존재하는 (정당한) 이유
무작정 낭비가 아니라 근거가 있음:

- **통합 파이프라인**: 같은 `camera_node`가 `vehicle.launch`(차→PC 스트리밍)에도, `onboard.launch`(온보드 계산)에도 쓰임. 출력 규격을 하나(`CompressedImage`)로 두면 코드 하나·버그 표면 최소·튜닝 일관성.
- **소비자 계약**: `autonomous_driving_node`, `viz_node`, `traffic_light`, `dash_line_tuner` 전부 `CompressedImage`를 구독. 규격 변경 비용이 큼.
- **DDS 전송 비용**: raw 640×480 BGR = ~900KB/프레임(30Hz → ~27MB/s). JPEG는 ~40-60KB로 **약 15-20배 작음**. localhost라도 DDS가 shared-memory를 안 쓰면 raw 전송이 오히려 더 느릴 수 있음.

→ **결론: `CompressedImage`를 쓰는 것 자체는 정당(CPU vs 대역폭 트레이드오프).**
   **하지만 USB MJPG를 굳이 풀었다가 다시 압축하는 왕복은 정당하지 않음** (MIPI(NV12·회전 필요)용 코드 경로를 USB에도 그대로 적용해서 생긴 부수효과).

### 해결 — USB MJPG passthrough
`camera_node.py` 수정: USB캠일 때 카메라가 이미 뱉는 **MJPG 프레임을 디코딩·재인코딩 없이 그대로 퍼블리시**.

- `cv2.VideoCapture(device, cv2.CAP_V4L2)` + `CAP_PROP_CONVERT_RGB=0` → raw JPEG 버퍼를 그대로 받음
- `timer_callback`에서 passthrough면 `imencode` 건너뛰고 원본 바이트 그대로 publish
- **안전 검증 3종** 통과 시에만 passthrough 활성화, 하나라도 실패하면 기존 디코딩+재인코딩 경로로 **자동 폴백**:
  1. 프레임이 JPEG SOI 마커(`0xFFD8`)로 시작하는가 (`CONVERT_RGB=0`이 먹혔는가)
  2. `cv2.imdecode`로 디코딩되는가 (소비자와 동일 함수 — Huffman 테이블 없는 MJPEG면 여기서 걸러짐)
  3. 해상도가 설정값(640×480)과 정확히 일치하는가
- 파라미터 `mjpg_passthrough`(기본 `true`)로 on/off. MIPI 경로는 손대지 않음.

### 결과
로그로 활성화 확인:
```
Camera capture opened with v4l2 MJPG passthrough (device=/dev/video1)
[camera source] : usb
```

CPU 실측(4초 창):
```
camera over 4s: total 95% of a core | user 94% | kernel 1%
```

- **디코딩·재인코딩은 실제로 제거됨** — 커널(USB 드라이버) 1%가 증거. USB에서 MJPG 읽는 건 거의 공짜.
- 남은 ~94%(user)는 **그 JPEG를 30fps로 ROS 토픽에 퍼블리시하는 `msg.data` 직렬화 비용** — passthrough든 재인코딩이든 동일하게 내던 고정 바닥값.
- passthrough로 camera 요구량이 내려갔지만 **전체 CPU%는 안 내려감** — CPU YOLO(`bisa_autonomous` ~170-183%)가 여유분을 즉시 흡수하기 때문. 온도 56°C로 스로틀 없음.

### 남은 레버 (이번엔 적용 안 함 — 카메라는 충분하다고 판단)
- **퍼블리시 fps 30 → 15**: camera 노드 비용 절반(~47%). 단 제어 루프 프레임이 절반이라 조향 반응이 살짝 느려질 수 있음. 카메라 하드웨어 fps도 같이 낮춰야 버퍼 지연 방지.

### 변경 파일
- `src/camera/camera/camera_node.py`
  - `import numpy as np` 추가
  - `mjpg_passthrough` 파라미터 + `self.passthrough` 플래그
  - `open_capture()`에 passthrough 우선 시도 로직
  - `open_mjpg_passthrough()` 신규 (검증 + 폴백)
  - `timer_callback()`에 passthrough 분기 (재인코딩 skip)

## 2. YOLO 추론 최적화 (NCNN export)

### 문제 — 진짜 병목은 CPU%가 아니라 추론 지연
카메라 최적화 중 발견:
```
YOLO infer: 900 ms/frame, 1.1 FPS effective (imgsz=320, hz_cap=4.0, device=cpu)
```
- 추론 한 번에 ~0.9초 → `inference_hz=4.0` 상한은 무의미, 실제 ~1 FPS.
- 자율주행 영향: 신호등을 **~1초 늦게** 인식(주행 중 위험), 표지판 투표 확정에 ~3초.
- 원인: `best.pt`(YOLOv8n급, 5.2MB)를 `ultralytics`가 **torch eager 모드**로 A72(ARM) CPU에서 실행. **torch 2.12.1+cpu 백엔드가 ARM 미최적화**.

### 해결 — NCNN export (imgsz 320 유지)
NCNN은 ARM 모바일 CPU 전용 추론 엔진. ultralytics 공식이 라즈베리파이/ARM에 추천.

작업:
1. `pip install ncnn` — ⚠️ **주의**: 이때 딸려온 의존성이 `numpy 1.26.4 → 2.2.6` 업그레이드 + `opencv-python 5.0`(pip) 설치를 해버림. numpy 2.x는 ABI 파괴적이고 pip-opencv엔 GStreamer가 없어 camera_node가 깨질 수 있음 → **`numpy==1.26.4` 재설치 + `pip uninstall opencv-python`으로 즉시 원복**. 시스템 cv2 4.5.4(GStreamer YES) 유지 확인.
   - 교훈: 이런 보드에선 `pip install <pkg> --no-deps` 또는 venv 격리 권장.
2. `YOLO("best.pt").export(format="ncnn", imgsz=320)` → `checkpoints/best_ncnn_model/` 생성 (원본 `best.pt`는 그대로).
3. `onboard.launch.py`의 `default_model_path()`를 `best_ncnn_model`로 변경 (온보드=CPU=NCNN, PC launch는 계속 `best.pt`/GPU).
4. `setup.py`에 NCNN 디렉토리 설치 추가 (`glob('checkpoints/*')`는 디렉토리를 못 복사 → 파일만 필터 + ncnn 디렉토리 별도 entry).

### 결과
| | 이전(torch .pt) | NCNN |
|---|---|---|
| 단독 벤치(imgsz 320) | 163 ms | **67 ms** (~2.4배) |
| **live 노드** | 900 ms / 1.0 FPS | **~500 ms / ~2 FPS** (~1.8배) |

NCNN 적용으로 live 추론 시간 반토막, 유효 FPS 2배. ✅

### ⚠️ 남은 병목 — 노드 내부 경합 (단독 67ms vs live 500ms, 7배 차이)
NCNN을 써도 live가 67ms까지 못 내려가는 이유. 두 가지가 겹침:

1. **CPU 여유 부족**: 4코어를 `bisa 168%` + `camera 100%` + (측정 중엔 `claude 68%`)가 나눠 씀. → **실주행 땐 원격 세션/불필요 프로세스 닫으면** 그만큼 YOLO로 환원.
2. **GIL 경합 (근본 원인)**: YOLO가 `bisa_autonomous_node` 안의 **파이썬 백그라운드 스레드**로 도는데, 메인 스레드가 카메라 프레임마다(passthrough로 30fps) lane 인식 + aruco를 numpy/opencv로 돌림. GIL 때문에 둘이 서로를 막아 YOLO 스레드의 전처리가 지연됨.

### 남은 레버 (효과 순, 미적용 — 선택 사항)
1. **검출기를 별도 프로세스로 분리** (가장 큰 한 방): YOLO를 `bisa_autonomous_node`에서 떼어 독립 프로세스/노드로 실행 → GIL 탈출, 전용 코어 확보. **~100-150ms(단독에 근접) 기대**. 대신 프로세스 간 프레임 전달(공유메모리/토픽) 구조 리팩터 필요.
2. **메인 스레드 부하율 낮추기**: aruco/lane 처리를 30fps → 더 낮게, 또는 카메라→노드 처리율 분리. 소규모 리팩터.
3. **imgsz 추가 축소** (320→256): 면적 0.64배로 추가 가속. 단 NCNN 모델을 256으로 **재export 필요**(현재 모델은 320 고정). 원거리 작은 신호등 감지력 소폭 하락.
4. **INT8 양자화**: 추가 ~2배 여지. 캘리브레이션 데이터 필요, 정확도 소폭 하락.

### 변경 파일
- `src/bisa/launch/onboard.launch.py` — `default_model_path()` → `best_ncnn_model` (imgsz 320 고정 주의)
- `src/bisa/setup.py` — checkpoints 파일 필터 + `best_ncnn_model/` 디렉토리 설치 entry
- `src/bisa/checkpoints/best_ncnn_model/` — 신규 (export 산출물: model.ncnn.param/bin, metadata.yaml)
- 환경: `ncnn 1.0.x` 설치 (numpy/opencv는 원복하여 시스템 유지)

### 참고 (imgsz 재export 방법)
```bash
python3 -c "from ultralytics import YOLO; YOLO('checkpoints/best.pt').export(format='ncnn', imgsz=256)"
# 이후 onboard.launch의 imgsz도 256으로 맞춰야 함 (NCNN은 export 해상도 고정)
```

## 3. `traffic_light_tuner` NCNN 적용 + 전체 노드 감사

### 문제 — 튜너만 아직 torch `best.pt`였음
SSH(`ssh -Y`, X11 forwarding)로 `traffic_light_tuner`를 노트북에 띄웠더니, 창은 뜨지만
카메라 영상이 몇 초씩 멈추고 데스크톱에 **"응답 없음 — 강제 종료/기다리기"** 대화상자가 뜸.
(사용자가 Force Quit을 눌러 `Killed`로 종료된 것이지, OOM이 아님.)

원인: 섹션 2의 NCNN 전환은 **`onboard.launch.py`의 `default_model_path()`만** 바꿨는데,
`traffic_light_tuner`는 **자기만의 `default_model_path()`** 를 따로 갖고 있어서 여전히
`checkpoints/best.pt`(torch, 900ms/frame)를 로드. YOLO 추론이 GUI 이벤트 루프를 굶겨서
(217% CPU) 창이 "응답 없음"으로 찍힘. 카메라는 정상(실측 17.3 FPS, 140KB/frame).

### 전체 노드 감사 — YOLO 로드 지점 3곳
`grep`으로 모델 로드 지점 전수 조사 결과:

| 노드 | 모델 결정 경로 | 적용 전 상태 |
|---|---|---|
| `autonomous_driving_node` | `onboard.launch`가 `model_path:=best_ncnn_model` 주입 | **이미 NCNN** ✅ |
| `traffic_light_tuner` | 자체 `default_model_path()` → `best.pt` | **미적용** ❌ (이번에 수정) |
| `dash_line_tuner`, `viz_node`, `param/power_gui`, `system_telemetry` | YOLO 미사용 (`CompressedImage`만 소비) | 해당 없음 |

→ NCNN이 안 먹은 실행 경로는 **튜너 하나**뿐이었음.

### 해결 — 튜너 `default_model_path()`를 NCNN 우선 + best.pt 폴백
`src/bisa/src/traffic_light.py`의 `default_model_path()` 수정:

```python
def default_model_path() -> str:
    ncnn = _find_package_file("checkpoints/best_ncnn_model", "")
    if ncnn:
        return ncnn
    return _find_package_file("checkpoints/best.pt", "checkpoints/best.pt")
```

- NCNN export 디렉토리가 있으면 그걸, 없으면 자동으로 `best.pt`로 폴백 (안전).
- 섹션 2에서 `setup.py`가 이미 `best_ncnn_model/`을 share에 설치하므로 추가 배포 작업 불필요.
- `colcon build --packages-select bisa`로 `install/`에 반영.

### 결과 — NCNN 로드 확인
```
Loading /home/topst/D-Racer-Kit/install/bisa/share/bisa/checkpoints/best_ncnn_model for NCNN inference...
[INFO] [bisa_traffic_light_tuner]: Detector loaded on device=cpu (task=detect)
```
튜너도 추론 900ms → ~500ms 경로로 전환, GUI "응답 없음" 해소.
`WARNING ⚠️ Unable to automatically guess model task, assuming 'task=detect'`는 무해
(NCNN metadata에 task 미명시 → `detect`로 정확히 폴백. 없애려면 로드 시 `task='detect'` 명시).

### 운영 메모 (SSH GUI 실행)
- OpenCV(GTK3)가 SSH X11에서 백엔드 초기화 실패 → **`GDK_BACKEND=x11`** 접두어로 실행:
  ```bash
  GDK_BACKEND=x11 ros2 run bisa traffic_light_tuner
  ```
- 접속은 `ssh -Y`(trusted). `-x`(소문자)는 X11 forwarding을 끄므로 주의.
- 튜너는 **imgsz 320 고정** 유지 (NCNN export가 320 고정이라 다른 값은 추론 깨짐).

### 잠재 버그 수정 — autonomous 폴백 model_path 오타 (`.pth` → `.pt`)
- 증상: `autonomous_driving_node`를 **런치 없이 `ros2 run`으로 직접** 실행하면, config 기본값이
  `dracer_config.py:86`의 `checkpoints/best.pth`(존재하지 않는 **`.pth`** 확장자, 실제 파일은 `best.pt`)
  라 모델을 못 찾아 **YOLO 검출이 조용히 비활성화**됨(노드는 안 죽고 차선주행만 됨). 실주행은 항상
  `onboard.launch`(NCNN 주입)로 도므로 그동안 무영향이었지만, 폴백 경로가 깨진 잠재 버그.
- 조치: 기본값을 존재하는 파일 `checkpoints/best.pt`로 정정 (**수정 완료**).
  - NCNN 강제는 하지 않음 — `device: "auto"`라 PC=GPU(best.pt)/보드=CPU 범용 폴백으로는 `best.pt`가 맞고,
    NCNN(CPU 전용)은 지금처럼 `onboard.launch`가 주입하는 역할 분담 유지.

### 변경 파일
- `src/bisa/src/traffic_light.py` — `default_model_path()` NCNN 우선 + best.pt 폴백 (+ docstring)
- `src/bisa/src/dracer_config.py` — detector 폴백 `model_path` 오타 정정 (`best.pth` → `best.pt`)

## 4. 튜너 신호등 인지 복구 — imgsz 버그 + GUI 프리즈 + onboard 실물 검증

섹션 3에서 튜너를 NCNN으로 바꾼 뒤 "튜너에서 신호등이 안 잡힌다 + 여전히 창이 응답 없음으로 뜬다"
증상이 남아 있었음. 두 개의 독립적인 버그였고 둘 다 수정 + 실물 검증 완료.

### 문제 A — 튜너가 신호등을 하나도 못 잡음 (검출 자체 실패)

원인: 섹션 2·3의 NCNN 전환에서 **imgsz를 320으로 내려주는 건 `onboard.launch.py`에만** 들어갔음
(`detector.imgsz:=320` 주입). 튜너는 launch를 안 거치고 `dracer_params.yaml`을 직접 로드하는데,
거기 `imgsz: 640`(PC GPU + best.pt 기준)이 그대로 살아있고 튜너는 `inference_hz`만 덮어씀.
**NCNN은 export 해상도(320) 고정**이라 640으로 돌리면 검출이 깨짐.

- 라이브 카메라 실측(동일 프레임 12장, 눈앞에 빨간불):
  | imgsz | 검출 |
  |---|---|
  | **320** (고정값) | `traffic_red` 0.91 — **12/12 프레임** |
  | 640 (버그) | **none — 0/12** |
- onboard 실주행 경로는 launch가 320을 주입하므로 원래부터 정상이었음. **튜너 직접-로드 경로만** 깨져 있었음.

해결 — `ncnn_export_imgsz()` 헬퍼 추가: NCNN 디렉토리의 `metadata.yaml`에서 고정 imgsz를 읽어
튜너 시작 시 `config.detector.imgsz`에 강제 반영 (onboard.launch의 `imgsz:=320` 역할을 직접-로드
경로에서 대신 수행). `best.pt`(torch)면 `None` 반환 → 아무것도 안 건드림. 나중에 256으로 재export해도
metadata를 따라 자동으로 맞춰짐. 로그: `NCNN model is fixed at imgsz=320; overriding config imgsz 640 -> 320`.

### 문제 B — 검출은 되는데 창이 "응답 없음 / 강제 종료" (GUI 프리즈)

원인: `rclpy.spin`은 **단일 스레드**. 그 위에서 `_render`(20Hz GUI 타이머)가 YOLO 추론
(`_detect_lights` → `predict`)을 **동기로 끝낸 뒤에야** `cv2.waitKey`를 호출. 즉 GUI 이벤트 펌핑이
매 프레임 추론(320이라도 CPU 경합 시 수백 ms~1s+)만큼 막힘 → WM이 "응답 없음"으로 판정.
imgsz를 320으로 낮춰 빨라져도 **추론이 GUI 스레드에 얹혀 있는 한** 근본 해결 안 됨.
(autonomous 노드는 이미 YOLO를 배경 스레드로 돌리는데 **튜너만** 안 그랬던 게 원인.)

해결 — 튜너도 YOLO를 **배경 스레드(`_infer_loop`)로 분리**:
- 배경 스레드가 검출 박스만 뽑아 락으로 공유. GUI 타이머는 **캐시된 박스만 그리고** `waitKey`를 20Hz로 계속 펌핑.
- 슬라이더가 튜닝하는 `classify_light`(색 분류, ~0.5ms)는 **GUI 스레드에 유지** → 슬라이더 즉각 반응.
- 배경 추론은 **~50% duty-cycle 스로틀**(`sleep(추론시간)`)로 CPU를 계속 물지 않게 해서 GUI 틱 경합 완화.
  튜너는 초당 1~2회 검출이면 충분.

- 라이브 실측(수정 후):
  | 지표 | 수정 전(인라인) | 수정 후(배경+스로틀) |
  |---|---|---|
  | GUI 키 펌프 간격 | 매 틱 ~540ms 블로킹 | 평균 **98ms(~10Hz)**, 최악 **616ms** |
  | 동일 시간 GUI 틱 수 | 31 | **122** (4배) |
- 최악 펌프 간격 616ms ≪ WM "응답 없음" 타임아웃(~5초) → **강제종료/대기 창 안 뜸.**

### onboard 실물 검증 (`ros2 launch bisa onboard.launch.py`)

실차 스택 기동 후 실물 신호등으로 red/green 양쪽 확인:

| 신호등 | camera | NCNN 검출 | `classify_light` 최종 판정 | 검출 토픽 |
|---|---|---|---|---|
| 🟢 초록불 | passthrough 17Hz | 박스 conf **0.93** | **green** ✅ | `/detect_green=True` |
| 🔴 빨간불 | passthrough 30Hz | 박스 conf **0.95** | **red** ✅ | `/detect_red=True` |

- NCNN live: 워밍업 첫 프레임 ~6.2s 후 **267ms/frame(~3.2 FPS)** 로 안정(hz_cap 4.0, imgsz 320, cpu).
- **YOLO 원시 라벨(`box.cls`)은 red/green 둘 다 `traffic_red`로 나옴** — 섹션 문서대로 신뢰불가
  (꺼진 빨강 렌즈도 빨강으로 보임). 최종 색은 `classify_light`의 픽셀 행 분석이 결정하며 둘 다 정확.
  예: 초록불에서 red_score 0.0008 vs green_score 0.269 → green / 빨간불에서 0.192 vs 0.080 → red.
- 영상 방향 정상(빨강 위/초록 아래) → `rotate-180` 정상 작동 확인. FSM은 `OUT_FINISH_STOP`,
  `throttle=0.00`으로 정지 유지(검증 중 차량 미이동), 확인 후 스택 정리.
